# Bonus 实验报告

## B1：全表扫描优化

**优化思路**

当前点查全部走 `SeqScan` 会导致 `stock`、`customer` 等大表每次查询都扫描大量页面。我们先先补全 B+ 树基础能力，再让优化器为点查和可利用索引前缀的查询生成索引扫描计划，使查询直接定位到较小的 B+ 树区间，减少扫描记录数和事务持锁时间。

**优化方法**

- 在 `Planner::get_index_cols()` 中选择可用前缀最长的索引，能使用索引时生成 `T_IndexScan`，否则才退化为 `T_SeqScan`。
- 补全 `IxNodeHandle` / `IxIndexHandle` 中的查找、插入、分裂、删除、合并、重分布等 B+ 树核心逻辑，保证索引可正确维护。
- 支持 `lower_bound`、`upper_bound` 和叶子节点扫描，为索引区间扫描提供基础接口。
- 在 `IndexScanExecutor` 中按复合索引列顺序构造完整的 `lower_key` 和 `upper_key`，支持多列等值条件和首个范围条件。
- 对完整等值查询使用 `lower_bound` 到 `upper_bound` 的扫描区间，避免全索引扫描后再过滤。
- 对 `UPDATE` 增加 `touches_index()` 判断，只维护被修改列涉及的索引。
- 构造索引 key 时优先使用栈缓冲区，减少频繁堆分配。
- 修复 B+ 树并发访问中的 page pin / unpin 问题，降低索引页被错误释放或重复访问的风险。

## B2：表级锁优化

**优化思路**

表级锁会让 NewOrder、Payment 等事务与任意扫描互斥，导致 wait-die 下大量事务 abort。我们的思路是把安全的点查场景从表级锁下沉到记录级锁，提高同表不同记录之间的并发度；范围扫描仍保留表锁以保证可串行化。

**优化方法**

- 引入多粒度锁，支持表级 `IS/IX/S/X/SIX` 和记录级 `S/X`。
- 索引点查读使用表级 `IS` + 记录级 `S`。
- 索引点查写使用表级 `IX` + 记录级 `X`。
- `InsertExecutor`、`UpdateExecutor`、`DeleteExecutor` 写路径只补表级 `IX`，具体记录冲突交给扫描阶段的记录锁处理。
- 只有完整等值索引查询使用记录锁；范围查询和前缀查询仍使用表级锁，避免幻读。
- 增加 `unlock_all(Transaction*)`，在 commit/abort 时批量释放事务持有的锁，降低锁管理器 latch 开销。

## B3：锁升级冲突优化

**优化思路**

当前的读后写场景中 S 锁升级 X 锁容易导致事务被强制 abort。我们的思路是完善锁管理器的等待队列和升级协议，让冲突按 Wait-Die 规则处理，而不是依赖固定超时或让多个升级者互相等待。

**优化方法**

- 普通锁请求进入 `request_queue_`，授予时同时检查已授予锁和排在前面的冲突请求。
- 每个锁对象增加 `upgrading_txn_` 和 `upgrading_mode_`，同一资源同一时间只允许一个事务升级。
- 如果已有事务正在升级，其他事务再尝试升级则触发 `UPGRADE_CONFLICT`。
- Wait-Die 按事务时间戳决策：年轻事务遇到更老的冲突事务时 abort，老事务等待年轻事务释放锁。
- 解锁和批量解锁时清理升级状态并唤醒等待队列。
- 在 TPC-C NewOrder 中对 `district` 热点行提前执行写路径，减少不必要的 `S -> X` 升级冲突。

## SQL 与执行表达能力扩展

**优化思路**

TPC-C 中常见的事务写法包含 `FOR UPDATE` 和自增/自减更新。如果 SQL 层不支持这些表达，事务只能通过更多读写语句绕开，容易制造额外锁升级和网络往返。

**优化方法**

- 在 parser 中增加 `SELECT ... FOR UPDATE` 支持，并将 `for_update` 标记传递到执行计划。
- 扩展 `SetClause`，支持 `col = col + value` 和 `col = col - value`。
- 在 `UpdateExecutor` 中直接执行整数和浮点数的增量更新，减少“先读旧值、客户端计算、再写回”的路径。
